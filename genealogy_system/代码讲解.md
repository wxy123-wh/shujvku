# 族谱管理系统代码讲解

本文档说明 `genealogy_system` 项目实现了哪些查询功能，以及这些功能分别对应到哪些代码文件。

## 1. 项目文件分工

| 文件或目录 | 作用 |
| --- | --- |
| `app.py` | Flask 主程序，包含登录、注册、族谱管理、成员管理、Dashboard、树形预览、祖先查询、亲缘关系查询等后端路由和 SQL 调用。 |
| `templates/` | HTML 页面模板，每个查询或管理功能都有对应页面。 |
| `static/style.css` | 页面样式。 |
| `sql/01_schema.sql` | PostgreSQL 建表脚本，定义表、主键、外键、CHECK 约束和触发器。 |
| `sql/02_indexes.sql` | 索引脚本，用于姓名模糊查询、父子关系查询、辈分统计、婚姻查询等优化。 |
| `sql/03_queries.sql` | 实验要求的核心 SQL 查询语句集中版。 |
| `sql/04_import.sql` | CSV 数据导入脚本。 |
| `sql/05_performance_compare.sql` | 有无索引性能对比用的四代后代递归查询。 |
| `scripts/generate_data.py` | 生成 10 个族谱、100000 条以上成员数据和关系数据。 |
| `scripts/check_data.py` | 校验 CSV 数据规模、外键引用、父母关系、婚姻关系等是否满足要求。 |
| `data/` | 生成后的 CSV 数据文件。 |

## 2. 数据库表对应关系

数据库结构主要在 `sql/01_schema.sql` 中定义。

| 表名 | 作用 | 主要关联 |
| --- | --- | --- |
| `users` | 系统用户表 | 用户注册、登录使用。 |
| `family_trees` | 族谱表 | 每个族谱有创建用户 `creator_user_id`。 |
| `tree_collaborators` | 族谱协作者表 | 表示用户可以参与哪些族谱。 |
| `members` | 成员表 | 每个成员属于一个族谱。 |
| `parent_child` | 父母子女关系表 | 用 `parent_id` 和 `child_id` 表示血缘关系。 |
| `marriages` | 婚姻关系表 | 用 `spouse1_id` 和 `spouse2_id` 表示配偶关系。 |

`sql/01_schema.sql` 还实现了两个触发器：

- `validate_parent_child()`：检查父母和子女是否属于同一族谱、父母性别是否正确、父母出生年份是否早于子女、父母辈分是否早于子女。
- `validate_marriage()`：检查婚姻双方是否属于同一族谱。

## 4. Web 系统中的查询功能说明

### 4.1 Dashboard 统计查询

对应文件：

- 后端：`app.py` 的 `dashboard()`
- 页面：`templates/dashboard.html`

实现内容：

- 查询当前登录用户能看到的族谱数量。
- 统计这些族谱下的总成员数。
- 按 `gender` 统计男性、女性人数。
- 在页面上显示男女比例。

核心逻辑：

- 先通过 `family_trees` 和 `tree_collaborators` 找出当前用户可访问的族谱。
- 再关联 `members` 表统计人数。
- 使用 PostgreSQL 的 `COUNT(...) FILTER (WHERE ...)` 分别统计男女数量。

### 4.2 族谱列表查询

对应文件：

- 后端：`app.py` 的 `trees()`
- 页面：`templates/trees.html`

实现内容：

- 登录后只查询用户自己创建的族谱，或者用户被邀请协作的族谱。
- 每个族谱显示谱名、姓氏、修谱时间、当前用户角色、成员数量。

核心逻辑：

- `family_trees.creator_user_id = 当前用户 ID` 表示自己创建。
- `tree_collaborators.user_id = 当前用户 ID` 表示受邀参与。
- 使用 `LEFT JOIN members` 统计每个族谱的人数。

### 4.3 成员列表和姓名模糊搜索

对应文件：

- 后端：`app.py` 的 `members()`
- 页面：`templates/members.html`
- 索引：`sql/02_indexes.sql`

实现内容：

- 显示某个族谱下的成员列表。
- 支持通过 `q` 参数按姓名模糊搜索。
- 支持 `page` 和 `page_size` 分页，避免一次加载 60000 条成员。
- 每个成员显示 ID、姓名、性别、生卒年、辈分、子女数量。

核心 SQL 特点：

- 使用 `m.name ILIKE %关键字%` 实现不区分大小写的模糊搜索。
- 使用 `LEFT JOIN parent_child pc ON pc.parent_id = m.member_id` 统计子女数量。
- `sql/02_indexes.sql` 中创建了：
  - `idx_members_tree_name`
  - `idx_members_name_trgm`

其中 `idx_members_name_trgm` 使用 PostgreSQL `pg_trgm`，用于优化 `%关键字%` 这种模糊查询。

### 4.4 树形预览后代分支

对应文件：

- 后端：`app.py` 的 `tree_preview()`
- 页面：`templates/tree_preview.html`
- 性能对比 SQL：`sql/05_performance_compare.sql`

实现内容：

- 输入一个根成员 ID。
- 用缩进列表展示该成员往下的后代关系。
- 可以设置查询深度，深度表示从根成员向下展开几层后代。
- 支持 `page` 和 `page_size` 分页，避免一次展开太多节点导致页面变慢。

核心逻辑：

- 使用 `WITH RECURSIVE descendants AS (...)` 从根成员开始递归。
- 每次通过 `parent_child.parent_id = 当前成员 ID` 找子女。
- 用 `path` 数组记录已经访问的成员，避免循环。
- 根据 `depth` 控制递归深度。

`sql/05_performance_compare.sql` 使用类似逻辑查询四代后代，用于实验中的有无索引性能对比。

### 4.5 祖先查询

对应文件：

- 后端：`app.py` 的 `ancestors()`
- 页面：`templates/ancestors.html`
- 实验 SQL：`sql/03_queries.sql` 第 2 条

实现内容：

- 输入成员 ID。
- 查询该成员的父亲、母亲，以及继续向上追溯的所有祖先。
- 返回祖先 ID、姓名、性别、生卒年、辈分、距离、父系或母系关系。
- 支持 `page` 和 `page_size` 分页。

核心逻辑：

- 使用 `WITH RECURSIVE ancestors AS (...)`。
- 第一层通过 `parent_child.child_id = 输入成员 ID` 找直接父母。
- 递归层继续用 `parent_child.child_id = 当前祖先 ID` 往上找祖父母、曾祖父母等。
- 使用 `path` 数组避免重复访问。

### 4.6 亲缘关系通路查询

对应文件：

- 后端：`app.py` 的 `relationship()`
- 页面：`templates/relationship.html`
- 实验 SQL：`sql/03_queries.sql` 第 6 条

实现内容：

- 输入两个成员 ID。
- 查询两者之间是否存在亲缘关系通路。
- 页面会先显示一句中文结论，再用节点箭头展示关系链，并列出逐步说明。
- 支持两类通路：
  - 直接婚姻关系。
  - 血缘共同祖先关系。

核心逻辑：

- `direct_spouse`：先判断两人是否直接存在婚姻关系。
- `up_from_a`：从成员 A 一直向上递归找祖先。
- `up_from_b`：从成员 B 一直向上递归找祖先。
- `blood_path`：寻找双方最近的共同祖先，并拼接 A 到共同祖先、共同祖先到 B 的路径。
- `best_path`：优先返回最短路径。
- `describe_relationship_steps()`：把路径中的每一条边翻译成中文，例如“夫妻”“父亲”“母亲”“儿子”“女儿”。
- `relationship_summary()`：生成页面顶部的中文结论。

之前如果直接在大族谱上做全图 BFS，容易因为分支过大而超时；现在代码采用“向上找共同祖先”的方式，更适合 10 万级成员数据。为了避免深代成员一次展开过多祖先路径，Web 查询把共同祖先追溯范围限制为 12 代；如果超过这个范围，页面会提示没有在 12 代追溯范围内找到共同祖先血缘通路。

## 5. 实验要求中的 5 个核心 SQL

这些 SQL 集中写在 `sql/03_queries.sql`。

| 实验要求 | SQL 位置 | 使用的表 |
| --- | --- | --- |
| 给定成员 ID，查询其配偶及所有子女 | 第 1 条 | `members`、`marriages`、`parent_child` |
| 给定成员 A 的 ID，查询所有祖先 | 第 2 条 | `members`、`parent_child` |
| 统计某个家族中平均寿命最长的一代人 | 第 3 条 | `members` |
| 查询年龄超过 50 岁且没有配偶的男性成员 | 第 4 条 | `members`、`marriages` |
| 找出出生年份早于该辈分平均出生年份的成员 | 第 5 条 | `members` |

另外，`sql/03_queries.sql` 第 6 条额外提供了亲缘关系通路查询，用于对应图形化系统中的亲缘查询功能。

## 6. 索引与性能优化

索引定义在 `sql/02_indexes.sql`。

| 索引 | 作用 |
| --- | --- |
| `idx_family_trees_creator` | 加速按创建用户查询族谱。 |
| `idx_tree_collaborators_user` | 加速查询用户受邀参与的族谱。 |
| `idx_members_tree_name` | 加速按族谱和姓名查询成员。 |
| `idx_members_name_trgm` | 使用 `pg_trgm` 优化姓名模糊搜索。 |
| `idx_members_tree_generation` | 加速按族谱、辈分分组统计。 |
| `idx_members_tree_gender` | 加速按性别统计或筛选。 |
| `idx_parent_child_parent` | 加速根据父节点查询子女。 |
| `idx_parent_child_child` | 加速根据子节点查询父母。 |
| `idx_marriages_spouse1` | 加速根据配偶 1 查询婚姻关系。 |
| `idx_marriages_spouse2` | 加速根据配偶 2 查询婚姻关系。 |

性能对比查询在 `sql/05_performance_compare.sql`：

- 用递归 CTE 查询某个成员的四代后代。
- 无索引时执行一次并截图。
- 执行 `sql/02_indexes.sql` 后再执行一次并截图。
- 对比 `EXPLAIN ANALYZE` 中的执行时间和扫描方式。

## 10. 总结

本系统的查询功能分为两类：

1. 图形化系统中直接使用的查询：
   - Dashboard 统计。
   - 族谱权限查询。
   - 成员列表和姓名模糊查询。
   - 树形后代预览。
   - 祖先递归查询。
   - 亲缘关系通路查询。

2. 实验报告和数据库验收用的 SQL 查询：
   - 配偶及子女查询。
   - 祖先递归查询。
   - 平均寿命最长辈分统计。
   - 50 岁以上无配偶男性查询。
   - 早于同辈平均出生年份成员查询。
   - 四代后代性能对比查询。

核心后端代码都在 `app.py`，实验 SQL 集中在 `sql/03_queries.sql`，索引和性能验证分别在 `sql/02_indexes.sql` 和 `sql/05_performance_compare.sql`。
