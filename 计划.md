## 3. 确定数据库表

先按下面这几张表做，够完成实验。

### 3.1 用户表 users

用途：保存注册用户。

字段建议：

- `user_id`：用户 ID，主键
- `username`：用户名，唯一
- `password_hash`：密码，简单版本可以先明文，但报告里建议写成密码哈希
- `created_at`：注册时间

检查项：

- [ ] 用户有唯一 ID
- [ ] 用户名不能重复
- [ ] 用户能登录

### 3.2 族谱表 family_trees

用途：保存每个族谱。

字段建议：

- `tree_id`：族谱 ID，主键
- `tree_name`：谱名
- `surname`：姓氏
- `revision_time`：修谱时间
- `creator_user_id`：创建用户，外键指向 `users.user_id`
- `created_at`：创建时间

检查项：

- [ ] 一个用户可以创建多个族谱
- [ ] 每个族谱必须有创建者

### 3.3 族谱协作者表 tree_collaborators

用途：记录哪些用户被邀请参与某个族谱。

字段建议：

- `tree_id`：族谱 ID，外键
- `user_id`：用户 ID，外键
- `role`：角色，例如 `owner`、`editor`

检查项：

- [ ] 创建者可以看到自己的族谱
- [ ] 被邀请的人也可以看到这个族谱

### 3.4 成员表 members

用途：保存家族成员基本信息。

字段建议：

- `member_id`：成员 ID，主键
- `tree_id`：所属族谱 ID，外键
- `name`：姓名
- `gender`：性别，建议 `M` 或 `F`
- `birth_year`：出生年份
- `death_year`：死亡年份，可以为空
- `generation`：第几代
- `biography`：生平简介

检查项：

- [ ] 每个成员属于一个族谱
- [ ] 支持同名同姓，因为用 `member_id` 区分
- [ ] 有 `generation` 字段方便统计辈分

### 3.5 父母子女关系表 parent_child

用途：保存父子、父女、母子、母女关系。

字段建议：

- `parent_id`：父或母的成员 ID，外键
- `child_id`：子女成员 ID，外键
- `relation_type`：关系类型，例如 `father`、`mother`

检查项：

- [ ] 一个成员可以有多个子女
- [ ] 一个成员最多有一个父亲和一个母亲
- [ ] 可以根据父节点 ID 快速查子节点

### 3.6 婚姻关系表 marriages

用途：保存配偶关系。

字段建议：

- `marriage_id`：婚姻 ID，主键
- `tree_id`：族谱 ID，外键
- `spouse1_id`：配偶 1，外键
- `spouse2_id`：配偶 2，外键
- `marriage_year`：结婚年份，可以为空

检查项：

- [ ] 给定成员 ID，可以查到配偶
- [ ] 配偶不能是自己

## 5. 写建表 SQL

文件：`sql/01_schema.sql`

要包含：

- [ ] `CREATE TABLE users`
- [ ] `CREATE TABLE family_trees`
- [ ] `CREATE TABLE tree_collaborators`
- [ ] `CREATE TABLE members`
- [ ] `CREATE TABLE parent_child`
- [ ] `CREATE TABLE marriages`
- [ ] 主键 `PRIMARY KEY`
- [ ] 外键 `FOREIGN KEY`
- [ ] `CHECK` 约束

建议写的 CHECK：

- [ ] 性别只能是 `M` 或 `F`
- [ ] 死亡年份为空，或者死亡年份大于等于出生年份
- [ ] 配偶不能是自己
- [ ] 父母子女不能是同一个人
- [ ] `generation` 大于等于 1

注意：老师例子说“父亲出生年份必须早于子女”，这个约束通常需要触发器才能严格做。不会写触发器也没关系，报告里可以说明数据生成时保证，数据库层用外键和基础 CHECK 保证主要一致性。

## 9. 做索引和性能对比

文件：`sql/02_indexes.sql`

必须覆盖两个需求：

- [ ] 根据姓名模糊查询
- [ ] 根据父节点 ID 查询子节点

建议索引：

```sql
CREATE INDEX idx_members_tree_name ON members(tree_id, name);
CREATE INDEX idx_parent_child_parent ON parent_child(parent_id);
CREATE INDEX idx_parent_child_child ON parent_child(child_id);
CREATE INDEX idx_members_tree_generation ON members(tree_id, generation);
```

如果 PostgreSQL 使用中文姓名模糊查询，普通 B-tree 对 `%某字%` 帮助有限。可以加 pg_trgm：

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_members_name_trgm ON members USING gin (name gin_trgm_ops);
```

性能对比步骤：

- [ ] 删除或暂时不建索引
- [ ] 执行“四代查询”，记录时间
- [ ] 截图 `EXPLAIN ANALYZE`
- [ ] 创建索引
- [ ] 再执行同一个“四代查询”，记录时间
- [ ] 截图 `EXPLAIN ANALYZE`
- [ ] 在报告里对比有无索引的时间差异
